## 一、目标问题与意义价值

### 1. 应用领域

- 1.1 劳动争议维权场景
  - 1.1.1 农民工欠薪维权
  - 1.1.2 试用期/实习生违法解除
  - 1.1.3 女职工"三期"特殊保护
  - 1.1.4 通用劳动争议（工伤认定、加班争议等）
- 1.2 目标用户群体
  - 1.2.1 劳动者（低法律素养、维权经验不足）
  - 1.2.2 基层法律援助人员
  - 1.2.3 高校法学/社工专业师生

### 2. 解决的核心问题

- 2.1 劳动者维权"三难"困境
  - 2.1.1 **不懂程序**：不知道仲裁/监察/诉讼的区别与流程
  - 2.1.2 **不会取证**：不知道哪些证据有用、如何组织证据链
  - 2.1.3 **不会写文书**：仲裁申请书/投诉书格式不规范、法条引用错误
- 2.2 现有工具的不足
  - 2.2.1 通用 AI 聊天缺乏场景引导，劳动者不知如何提问
  - 2.2.2 单点工具（计算器、模板下载）割裂，无法形成闭环
  - 2.2.3 法条检索结果堆砌，缺乏针对案情的分析整合

### 3. 实现目标与基本功能

- 3.1 总体目标
  - 3.1.1 构建"场景 → 案件 → Pipeline → 材料包"全流程闭环
  - 3.1.2 实现"场景找用户、系统配路径"的智能服务模式
- 3.2 核心功能模块
  - 3.2.1 场景化 Intake（结构化表单 + 自由描述双入口）
  - 3.2.2 智能建案与维权路径规划（LEAP）
  - 3.2.3 多模态证据整理与质证（MICE）
  - 3.2.4 混合检索法律研究引擎（HERA）
  - 3.2.5 结构化文书生成引擎（SADG）
  - 3.2.6 法律工具箱（检索、计算器、合同审查、企业查询等）
  - 3.2.7 办事资源出站（31 省官方平台 + 12348）

### 4. 理论意义与应用价值

- 4.1 理论意义
  - 4.1.1 提出"场景驱动智能服务设计（SDAD）"范式
  - 4.1.2 探索 LLM Agent 在垂直法律场景的落地路径
  - 4.1.3 验证混合 RAG + 多模态 OCR 在证据场景的有效性
- 4.2 应用价值
  - 4.2.1 降低劳动者维权门槛，提升维权效率
  - 4.2.2 辅助基层法律援助人员，缓解法律资源不均
  - 4.2.3 为"AI + 法律"垂直应用提供可复现的技术方案

## 二、设计思路与方案

### 1. 总体设计思路

- 1.1 "场景驱动"核心理念
  - 1.1.1 与传统 LegalTech 的对比（表格：用户找工具 vs 系统配路径）
  - 1.1.2 场景化设计的优势：降低认知门槛、提升完成率
- 1.2 技术路线选择
  - 1.2.1 为什么选 LangChain + LangGraph（而非直接调 API / 纯规则引擎）
  - 1.2.2 为什么用混合检索（而非纯向量 / 纯关键词）
  - 1.2.3 为什么需要多 Agent 协作（而非单 Agent / 单轮对话）

### 2. 系统架构设计

- 2.1 六层体系架构
  - 2.1.1 用户层 → 展现层 → 业务层 → 场景层 → 技术层 → 数据层
  - 2.1.2 各层职责与交互关系（配图）
- 2.2 技术栈选型
  - 2.2.1 后端：FastAPI + SQLAlchemy + Redis
  - 2.2.2 前端：React 19 + TypeScript + Tailwind
  - 2.2.3 AI 层：LangChain + LangGraph + ChromaDB + BM25
  - 2.2.4 文档处理：pypdf + PyMuPDF + qwen-vl-ocr
- 2.3 模块依赖关系（配依赖图）

### 3. 关键算法设计

- 3.1 LEAP — 语言增强型维权路径规划
  - 3.1.1 输入：场景 ID + 结构化答案 / 自由文本
  - 3.1.2 处理：STDE 槽位填充 → 案由匹配 → Agent 评估
  - 3.1.3 输出：next-step + pipeline_tasks + 路由 prefill
  - 3.1.4 与 LangGraph Supervisor 的集成
- 3.2 MICE — 多模态证据一致性质检引擎
  - 3.2.1 感知层：VL-OCR + PyMuPDF 双引擎
  - 3.2.2 抽取层：劳动领域 regex + 词典 NER
  - 3.2.3 校验层：时间线/金额/主体规则
  - 3.2.4 对照层：case_readiness 清单 + 就绪度评分
- 3.3 HERA — 混合集成检索架构
  - 3.3.1 向量检索：ChromaDB + all-MiniLM-L6-v2
  - 3.3.2 关键词检索：BM25Okapi + jieba 分词
  - 3.3.3 融合算法：RRF（Reciprocal Rank Fusion）
  - 3.3.4 场景化权重配置（法条 0.7/0.3、知识库 0.6/0.4）
- 3.4 SADG — 结构化 Agent 文书生成
  - 3.4.1 模板约束 + LLM 增强 + 法条引用校验
  - 3.4.2 LCEL Pipeline：解析 → 检索 → 生成 → 审查 → 导出
  - 3.4.3 支持 17+ 文书类型

### 4. 多 Agent 协作设计

- 4.1 LangGraph Supervisor 状态机
  - 4.1.1 状态定义（SupervisorState）
  - 4.1.2 图结构（START → supervisor → END）
  - 4.1.3 降级策略（LangGraph → 规则驱动）
- 4.2 五个专家 Agent
  - 4.2.1 GuidanceAgent：维权指引
  - 4.2.2 EvidenceAgent：证据整理
  - 4.2.3 DocgenAgent：文书生成
  - 4.2.4 ResearchAgent：案情分析
  - 4.2.5 RecordsAgent：记录汇总
- 4.3 Agent 评估与调度逻辑

### 5. 数据流与 Pipeline 设计

- 5.1 LCEL 可组合 Pipeline
  - 5.1.1 研究 Pipeline：分解 → 检索 → 合成 → 深度分析
  - 5.1.2 文书 Pipeline：解析 → 检索 → 生成 → 审查 → 导出
  - 5.1.3 PipelineStatsCallback 监控
- 5.2 端到端数据流（配时序图）

## 五、创新与特色

### 1. 场景驱动智能服务设计（SDAD）

- 1.1 提出"场景找用户、系统配路径"的服务范式
- 1.2 三大专项场景 + 通用入口的场景矩阵
- 1.3 五段式闭环（情境采集 → 智能建案 → 证据就绪 → 策略文书 → 办事出站）
- 💡 **填充思路**：与传统 LegalTech 对比表，突出"场景化"差异

### 2. 四大命名算法体系

- 2.1 LEAP：语言增强型维权路径规划
- 2.2 MICE：多模态证据一致性质检引擎
- 2.3 HERA：混合集成检索架构
- 2.4 SADG：结构化 Agent 文书生成
- 💡 **填充思路**：每个算法一句话定义 + 核心技术点 + 解决的问题

### 3. LangChain + LangGraph 工业级实现

- 3.1 LCEL 可组合 Pipeline（查询分解 → 并行检索 → 报告合成 → 深度分析）
- 3.2 LangGraph Supervisor 状态机 + 5 专家 Agent
- 3.3 PipelineStatsCallback 全链路监控
- 💡 **填充思路**：强调不是 demo 级调用，而是生产就绪的编排架构

### 4. "三源融合"法律资源整合

- 4.1 官方办事资源（可信出站）
- 4.2 本地向量知识库（可检索）
- 4.3 大模型 synthesis（可读写）
- 💡 **填充思路**：解决劳动者"去哪办、查什么、写什么"三件事

### 5. 向后兼容与渐进式降级

- 5.1 LangGraph → 规则驱动调度降级
- 5.2 混合检索 → 纯向量检索降级
- 5.3 双协议 LLM 自动切换
- 💡 **填充思路**：强调系统的鲁棒性和工程成熟度
